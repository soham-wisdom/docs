title: "Build an Agent in Wisdom AI"
---

Agents in Wisdom AI are automated workflows that analyze your data on a schedule and deliver results to you, without requiring manual queries. This guide walks through every step of building an agent, from choosing a build mode to testing and deploying.

## Before you start

Agents are built inside the Agent Builder, accessible from the **Agents** section in the top navigation. Once inside, you'll choose between two build modes:

- **Visual Mode**: a drag-and-drop node canvas for building multi-step, branching logic.
- **Prompt Mode:** a natural language interface where you describe what you want, and AI generates the agent for you.

Both modes produce the same type of agent and share the same components: tools, skills, triggers, a variable store, a Run Visualizer, and a shared lifecycle (Draft → Test → Publish → Activate → Run).

If you're new to agents, Prompt Mode is the fastest way to get started. Visual Mode gives you more control over branching and conditional logic.

This article covers the visual mode creation. Read Create an Agent to learn how to do it using the prompt mode.

## Build your agent



### Step 1: Write Your Instruction

The instruction is the most important part of your agent. It tells the agent what to analyze, when to flag something, and how to format the output.

A strong instruction covers three things:

1. **What to analyze** — the data, metrics, or entities to look at
2. **What conditions matter** — thresholds, comparisons, or time ranges
3. **What to include in the output** — format, sections, tables, and edge cases

#### Weak vs. strong instructions

| Weak | Strong |
| --- | --- |
| "Fetch open issues from the tracker" | "Fetch all issues labeled 'urgent' created in the last 14 days. Exclude issues with status 'Closed' or 'Duplicate'. Group by assignee. For each assignee include: issue ID, title, creation date, and current status. Sort each group by creation date descending." |
| "Analyze the data for trends" | "Compare this period's metrics against the prior period. Flag any category where the value changed by more than 20%. For flagged categories, note whether it is an increase or decrease and compute the exact percentage change. If no category exceeds the threshold, state that all metrics are stable." |
| "Create a summary report" | "Create a report with the heading 'Weekly Service Health Report'. Section 1: Overview — a bulleted list with total services monitored, services with incidents, and overall uptime (X.XX% format). Section 2: Incidents (only if any occurred) — for each affected service, list incident ID, title, severity, and duration. Section 3: Summary — if no incidents, state 'All services healthy.'" |

Make sure your instruction also covers edge cases: what should the agent say when there's no data, no anomalies, or nothing to flag?

### Step 2: Set a Trigger (Optional)

Triggers control when your agent takes action vs. stays silent. If you only want to hear about problems — anomaly detection, threshold breaches, SLA violations — add a trigger. If you want regular reporting regardless of what the data shows (daily summaries, weekly reviews), skip the trigger and let the agent always run.

#### Use concrete thresholds

| Vague | Concrete |
| --- | --- |
| "If sales are low" | "If daily revenue drops by more than 10% compared to the previous 7-day average" |
| "If there are problems" | "If the error rate exceeds 5% or if any orders are stuck in pending status for more than 48 hours" |
| "If something changed" | "If revenue falls outside the 4% monthly variance range" |

Start with a single condition. Once you confirm it fires correctly, add more. Debugging a multi-condition trigger from scratch is significantly harder than layering conditions one at a time.

### Step 3: Choose a Schedule

Match the schedule to how often your data changes and how urgently you need to act on it.

| Frequency | Best for | Examples |
| --- | --- | --- |
| Hourly | Urgent, real-time monitoring only | Production error rate spikes, critical SLA breaches |
| Daily | Most monitoring tasks (recommended default) | Sales metrics, pipeline health, support ticket volume |
| Weekly | Trend analysis and summaries | Week-over-week performance, team productivity, campaign results |
| Monthly | High-level reviews and strategic reporting | Monthly business review, quarterly forecasts, budget tracking |

A daily schedule is the right default for most agents. Only use hourly if you genuinely need to react within the hour — frequent runs add noise and make triggers harder to tune.

### Step 4: Specify Output Format

If you don't tell the agent how to present its results, the output will be inconsistent. Be explicit.

- Name your section headings: "Section 1: Overview", "Section 2: Details"
- Specify table columns: "Include columns: Deal Name, Stage, Days Stuck, Owner"
- Define number formats: "Show currency as \$X,XXX" and "Show percentages as X.X%"
- Handle conditional sections: "Only include the Incidents section if incidents occurred"
- Limit volume: "Limit to top 10 entries ranked by largest change"

For agents with personalization enabled, you can address the reader directly in the output:

> "Open with a 2-3 sentence summary addressing the reader, e.g.: 'This week your team handled X items averaging Y per day — up/down Z% vs. your 4-week average.'"

### Step 5: Build in Visual Mode (if applicable)

If you're using Visual Mode, additional principles apply at the node level.

#### Node instructions should be self-contained

Each node's instruction must make sense on its own. Don't reference other nodes by name. Instead, describe the input data the node should expect:

> "Given the list of assignees from the incoming issues, fetch their last 10 resolved tickets."

#### Describe outcomes, not procedures

Tell the node what you want, not how to implement it.

| Procedural (avoid) | Outcome-focused (preferred) |
| --- | --- |
| "SELECT \* FROM bugs WHERE severity = 'critical' AND created\_at \> NOW() - INTERVAL 7 DAY" | "Fetch all critical bugs from the last 7 days, grouped by component" |
| "Use the EVALUATE\_CRITERIA tool to check if error count exceeds threshold" | "Check if the total number of errors exceeds 10" |

#### One responsibility per node

A node can do complex work, but that work should be cohesive. If an instruction covers two unrelated concerns, split them into separate nodes.

#### Use concrete values in If/Else conditions

| Vague | Concrete |
| --- | --- |
| "There are many errors" | "The total count of items with status 'error' is greater than 10" |
| "Performance is poor" | "The average response time exceeds 500ms" |

#### Route data through Control Nodes

If data is needed downstream after an If/Else node, connect it through the node rather than short-circuiting around it. Bypassing the If/Else node causes both branches to activate regardless of the condition.

#### Artifact Builder tips

- Connect data nodes directly to the Artifact Builder if the output needs tables or charts — don't route everything through a Summary node first
- Split the Artifact Builder's instruction into two parts: one for how to structure the data (grouping, lists, tables), and one for how to format the final output (headings, layout, number formats)
- Be explicit about the output format — specify headings, bullet patterns, and number formats exactly as you want them to appear

### Step 6: Test Before Deploying

Always test your agent before activating it. A Preview feature is coming soon that will let you test agent outputs directly from the editor. In the meantime, use test runs to validate behavior.

#### Iterate based on results

- Output too broad → add filtering criteria
- Important data missing → name the specific fields you need
- Inconsistent formatting → add explicit formatting instructions
- Trigger fires too often → tighten the threshold
- Trigger never fires → loosen the threshold

#### Use Deep Analysis for complex tasks

Enable Deep Analysis when the agent needs to explore multiple aspects of the data, cross-reference different metrics, or produce comprehensive insights. It takes longer but yields more thorough results. For simple metric monitoring, leave it off.

### Deployment Checklist

Before activating your agent, confirm:

- Instruction covers what to analyze, what thresholds matter, and how to format the output
- Trigger criteria use concrete values (if applicable)
- Schedule matches how often the data changes
- Output format specifies sections, columns, and number formats
- Edge cases are handled (what to say when nothing is flagged)
- Trigger tested independently before deploying
- Full end-to-end test run completed
- Delivery channel configured (email, Slack, or webhook)

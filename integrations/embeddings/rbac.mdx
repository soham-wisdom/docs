---
title: "RBAC for Embedded Users"
---

Embedded users inherit WisdomAI's role-based access control (RBAC) system. The role you assign when provisioning a user controls what they can see and do inside the embedded interface. There is no separate embed-specific permission layer.

<Note>
  Custom roles are not yet supported. You must use one of the built-in roles listed below.
</Note>

## How roles affect embedded views

Visibility of edit controls, dashboard creation buttons, and AI workstreams is determined by the user's role assignment. To **hide edit controls in an embedded dashboard**, assign the user the **Viewer** role.

The table below summarizes the most common roles used in embedded deployments.

| Role | Dashboard read | Dashboard creation | Edit SQL | AI workstream |
| --- | :-: | :-: | :-: | :-: |
| `ADMIN` | Yes | Yes | Yes | Yes |
| `VIEWER` | Yes | No | No | No |
| `BASIC_EXPLORER` | No | Yes | No | No |
| `EXPLORER` | No | Yes | Yes | Yes |

<Note>
  The **Viewer** role is pure read-only and cannot create or modify any resources.
</Note>

## Assigning a role at provisioning time

Pass the specific `roleId` in the `roleAssignments` array when calling `createUsers`. Role IDs are unique UUIDs.

<Note>
  For a full list of **role IDs** to use in `roleAssignments`, see the [Roles reference](/integrations/embeddings/graphql-api/objects/roles).
</Note>

```bash
curl -X POST \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access_token>" \
  -d '{
    "query": "mutation CreateUsers($input: CreateUsersInput!) { createUsers(input: $input) { status { code message } } }",
    "variables": {
      "input": {
        "emails": ["alice@yourcompany.com"],
        "workspaceID": "workspace_123",
        "roleAssignments": [
          {
            "roleId": "00000000-0000-0000-0000-000000000003",
            "scopes": ["domain_456"]
          }
        ]
      }
    }
  }' \
  https://{ACCOUNT}.wisdom.ai/graphql
```

## Roles & Permission Reference

The tables below show the full permission breakdown across all roles and resource types. Use these as a reference when deciding which role to assign for a given use case.

<Note>
  In the API, **Story** refers to **Dashboards**. Roles labeled **Writer** grant edit access to existing resources but do not grant permission to create new ones.
</Note>

<Accordion title="Role scope & capabilities">
  | Role | Scope | Dashboards | Chat / Q&A | SQL editing | AI workstream | Create resources |
  | --- | --- | :-: | :-: | :-: | :-: | :-: |
  | `SITE_ADMIN` | Global | Read, create, clone | Create | Yes | Yes | Yes |
  | `ADMIN` | Organization | Read, create, clone | Create | Yes | Yes | Yes |
  | `DATA_ADMIN` | Domain | Read, create, clone | Create | Yes | Yes | Yes |
  | `CREATOR` | Any | No | No | No | No | Domains, connections |
  | `EXPLORER` | Domain | No | Create | Yes | Yes | No |
  | `BASIC_EXPLORER` | Domain | No | Create | No | No | No |
  | `VIEWER` | Organization | Read only | Read only | No | No | No |
  | `OBSERVER` | Organization | No | No | No | No | No |
  | `STORY_VIEWER` | Dashboard ID | Read one | No | No | No | No |
  | `STORY_WRITER` | Dashboard ID | Edit/clone one | No | No | No | No |
  | `AGENT_VIEWER` | Agent ID | No | No | No | No | No |
  | `AGENT_WRITER` | Agent ID | No | No | No | No | No |
  | `SCHEDULE_VIEWER` | Schedule ID | No | No | No | No | No |
  | `SCHEDULE_WRITER` | Schedule ID | No | No | No | No | No |
  | `CONNECTION_ADMIN` | Connection | No | No | No | No | No |
</Accordion>

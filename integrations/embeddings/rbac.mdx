title: "RBAC for Embedded Users"
---

Embedded users inherit WisdomAI's role-based access control (RBAC) system. The role you assign when provisioning a user controls what they can see and do inside the embedded interface. There is no separate embed-specific permission layer.

## How roles affect embedded views

Visibility of edit controls, dashboard creation buttons, and AI workstreams is determined by the user's role assignment. To **hide edit controls in an embedded dashboard**, assign the user the **Viewer** role.

| Role | Edit SQL | Dashboard creation | AI workstream |
|------|:---:|:---:|:---:|
| `VIEWER` | No | No | No |
| `BASIC_EXPLORER` | No | Yes | No |
| `EXPLORER` | Yes | Yes | Yes |

*Note: The **Viewer** role is pure read-only and cannot create or modify any resources.*

## Assigning a role at provisioning time

Pass the specific `roleId` in the `roleAssignments` array when calling `createUsers`. Role IDs are unique UUIDs. You should fetch the current IDs for your environment via the `roles` query before provisioning.

```bash
curl -X POST \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access_token>" \
  -d '{
    "query": "mutation CreateUsers($input: CreateUsersInput!) { createUsers(input: $input) { status { code message } } }",
    "variables": {
      "input": {
        "emails": ["alice@yourcompany.com"],
        "workspaceID": "workspace_123",
        "roleAssignments": [
          {
            "roleId": "00000000-0000-0000-0000-000000000003",
            "scopes": ["domain_456"]
          }
        ]
      }
    }
  }' \
  https://{ACCOUNT}.askwisdom.ai/graphql
```

## Complete Role & Scope Reference

Roles in WisdomAI must be scoped to specific resource IDs. A role assigned with an invalid or missing scope will result in the user having no active permissions.

| Role | Required Scope | Purpose |
|------|----------------|---------|
| `SITE_ADMIN` | Global | Full access across all organizations and workspaces |
| `ADMIN` | Organization | Full organization-level management |
| `DATA_ADMIN` | Domain | Full domain access (cannot manage IAM or Connections) |
| `EXPLORER` | Domain | Full query access including SQL editing and AI workstreams |
| `BASIC_EXPLORER` | Domain | Query data without SQL or AI workstream access |
| `VIEWER` | Organization | Read-only access across dashboards, chats, and agents |
| `OBSERVER` | Organization | Minimal read-only access (IAM/Workspace only; no dashboards) |
| `CREATOR` | Any | Permission to create domains and connections only |
| `CONNECTION_ADMIN`| Connection | Manage (Read/Write/Delete) one specific data connection |
| `STORY_VIEWER` | Dashboard ID | View a specific dashboard |
| `STORY_WRITER` | Dashboard ID | Edit/Clone a specific dashboard (cannot create new ones) |
| `AGENT_VIEWER` | Agent ID | View a specific AI agent |
| `AGENT_WRITER` | Agent ID | Edit a specific AI agent (cannot create new ones) |
| `SCHEDULE_VIEWER` | Schedule ID | View a specific report schedule |
| `SCHEDULE_WRITER` | Schedule ID | Edit a specific report schedule (cannot create new ones) |

*Note: In the API, **Story** refers to **Dashboards**. Roles labeled **Writer** grant edit access to existing resources but do not grant permission to create new ones.*
